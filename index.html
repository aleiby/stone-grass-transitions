The Problem

Bilinear blending: smooth gradient
ignores texture features

Crack-following: boundary traces
natural seams in the stone

Stone textures have natural cracks between individual stones. If we detect those cracks and route transition boundaries through them, the stone-grass edge looks like grass growing between the stones — not an arbitrary line painted across them.

Surface Case: Which Corners Are Stone?

Within each grass tile, corners can be stone or grass. The 4-bit SurfaceCase bitmask determines how many and which edge paths are needed:

  • 0 paths: all-grass (0000) or all-stone (1111)
  • 2 paths: one corner differs (corner/three cases)
  • 4 paths: edge, saddle, or mixed cases

Each edge path connects the centers of adjacent tiles. For mixed tiles, these paths will be routed through the crack network in texture space.

// Per-instance data packed into INSTANCE_CUSTOM
float surfaceBits  = INSTANCE_CUSTOM.x; // 4-bit mask
float surfaceRot   = INSTANCE_CUSTOM.y; // canonical rotation
float crackPathRow = INSTANCE_CUSTOM.z; // atlas row

float crack_depth = h_blur - h_pixel;                      // positive = in a crack
float in_crack = smoothstep(midpoint - spread/2, midpoint + spread/2, crack_depth);

Step 2: Center Snapping

From geometric to crack center

Each tile's geometric center (centroid of its 4 corners) may land on solid stone. We search the local neighborhood for the nearest crack pixel.

Two-tier scoring: Any crack pixel beats any stone pixel (cost 0 vs 1000). Among crack pixels, prefer the closest to the geometric center. This ensures paths start and end inside the crack network.

float score = is_stone * 1000.0 + dist;
// crack at distance 0.9 beats stone at distance 0.01

Step 3: Edge Pathfinding — Monotonic DP

Rotated-frame monotonic sweep

For each edge (pair of adjacent tile centers), we sweep 65 steps along the center-to-center vector with 64 perpendicular lanes.

The cost field makes cracks cheap and stone expensive. The DP finds the minimum-cost path constrained to advance monotonically along the tangent.

// Per lane: check ±max_step neighbors
for (int dr = 0; dr <= max_step; dr++) {
  float candidate = dp_prev[from]
    + cell_cost + dr * diagonal_penalty;
  if (candidate < best) {
    best = candidate;
    best_from = from;
  }
}

Step 4: Flap Folding — The Sequential Overwrite Classifier

Why not simple 2D quadrants?

Crack paths are curved and at arbitrary angles. A fixed quadrant split (NE/SE/SW/NW) fails when paths tilt. Instead, we process edges S➔E➔N➔W, each one overwriting the classification for pixels on its positive side.

The algorithm

For each pixel, evaluate all 4 edge paths. The last edge whose positive side contains this pixel (with domain guards) determines which corner owns it.

// sector_corner maps edge → corner
// S→BR, E→TR, N→TL, W→BL
int sector_corner[4] = {1,2,3,0};
int assigned = 0; // start with S edge
for (int i = 1; i < 4; i++) {
  if (side[i] <= 0) continue;
  // domain + wrap guards...
  assigned = i;
}
corner = sector_corner[assigned];

The wrap guard prevents W from stealing pixels that S legitimately owns when they run nearly parallel.

Step 5: Per-Patch Mask Generation

Raw classification

After Gaussian blur

Applied to stone texture

The classify shader writes an R8 texture in stone-texture UV space (not screen space). Each texel finds its owning tile via point-in-quad, runs the sequential overwrite classifier, and writes the corner's stone flag. A separable Gaussian blur softens the transition. The fragment shader samples with a single texture() call.

Vertex Shader: Data Carrier Transform

// MODEL_MATRIX encodes tile corners, not a transform
vec2 c0 = vec2(MODEL_MATRIX[0][0], MODEL_MATRIX[0][1]);
vec2 c1 = vec2(MODEL_MATRIX[0][2], MODEL_MATRIX[1][0]);
vec2 c2 = vec2(MODEL_MATRIX[1][1], MODEL_MATRIX[1][2]);
vec2 c3 = vec2(MODEL_MATRIX[2][0], MODEL_MATRIX[2][1]);

// Bilinear interpolation for irregular quads
vec2 world_xz = mix(
  mix(corners[0], corners[1], pos.x),
  mix(corners[3], corners[2], pos.x),
  pos.y);

Cliff Normals via Jacobian

// J^{-T} transforms normals through deformation
vec2 dWdx = mix(c1-c0, c2-c3, pos.y);
vec2 dWdy = mix(c3-c0, c2-c1, pos.x);
float det = dWdx.x*dWdy.y - dWdx.y*dWdy.x;

Fragment Shader: Compositing

// 1. Sample the precomputed crack mask
float stone_raw = texture(crack_mask_tex, mask_uv).r;
float stone = smoothstep(0.2, 0.8, stone_raw);

// 2. PBR stone with parallax occlusion
vec2 pom_uv = apply_pom(stone_uv, view_ts, depth);
vec3 albedo = textureLod(stone_albedo, pom_uv, mip);

// 3. Moss in cracks (local contrast again)
float moss = in_crack * shade_factor * strength;
albedo = mix(albedo, moss_color, moss);

// 4. Blend with grass
vec3 final = mix(grass_color, albedo, stone);

Performance & Implementation Notes

Compute Budget

  • Center snapping: 1 workgroup per tile, 64 threads. ~17×17 search grid per tile = 289 height map samples.
  • DP pathfinding: 1 workgroup per edge, 64 threads. 65 steps × 64 lanes = 4160 cost evaluations per edge.
  • Classification: 8×8 workgroups over mask texture. Point-in-quad test iterates all tiles (use spatial acceleration for large patches).
  • Blur: Separable Gaussian, radius 4. Two passes.
  • Typical patch: ~50 tiles, ~80 edges, 1024×1024 mask → <2ms total GPU time

Memory

  • Edge paths: 65 × edges, R8 = ~5KB per patch
  • Edge metadata: 2 × edges, RGBA32F = ~2.5KB
  • Crack mask: 1024×1024, R8 + mips = ~1.3MB per patch

Integration Checklist

  • Need a stone heightmap with visible cracks (most stone PBR textures work)
  • Need per-tile corner data: 4 world-space positions + stone/grass flags
  • Need adjacency: which tiles share edges (for edge list)
  • Compute pathfinding once at terrain load time (not per-frame)
  • Fragment shader cost: 1 texture sample + smoothstep (negligible)

Alternative: Wavefront Relaxation

For higher quality, replace DP with a full shortest-path solve via iterative relaxation on a corridor atlas. Trades the 64-lane band limit for texel-resolution paths at the cost of multiple relaxation passes.

The DP approach is sufficient for most use cases. The wavefront variant is only needed when paths must traverse very wide corridors at shallow angles.
